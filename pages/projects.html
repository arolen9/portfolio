Deploying a Honeypot

Objectives:

  • To create a honeypot web server that logs intrusion attempts
  • To observe attacker behavior and learn who’s probing the network service (ordinary users wouldn’t visit it)

Methodology & Tools:

  • Accessed Kali Linux and Windows Server 2022 virtual machines using provided credentials.
  • Opened a terminal in Kali and retrieved IP address using ip addr.
  • Launched PenTBox with root privileges using sudo ./pentbox-1.8/pentbox.rb.
  • Selected “Network tools → Honeypot” from the PenTBox menu.
  • Under “Manual Configuration” entered port 80 and entered the message: "You shouldn't be here!" as the fake site content.
  • Enabled logging and confirmed default log name.
  • Navigated to Kali’s fake Web Server page using Firefox on the Windows Server.
  • Verified honeypot response and intrusion detection logs in Kali’s terminal.

Outcome & Key Learnings:

  • Demonstrated how honeypots can detect unauthorized access attempts.
  • Showed how simple deception can reveal probing behavior from potential attackers.
  • Reinforced the importance of monitoring unused or unexpected services.
  • Highlighted the value of logging and analyzing intrusion data for network security.

    Exploiting and Hardening Windows

    Objective: To demonstrate the importance of securing an operating system.

    Methodology & Tools:

    • From Kali Linux attacking machine ran Nmap to scan Windows Server to determine open ports and services, revealing vulnerabilities like an open FTP (port 21) and HTTP (port 80).
    • Used FTP command to connect to the target FTP server to test for anonymous access and file upload/download capabilities.
    • Used Metasploit Penetration Testing Framework to exploit the Windows Server using a known vulnerability, specifically the MS17-010 (SMB) exploit, then smbclient command to enumerate file shares on the Windows Server to test for unauthenticated shares.
    • Post-exploitation, used net user command to add an unauthorized administrative account.
    • Removed the Web Server (IIS) role, which included the insecure default website and anonymous FTP service, using Server Manager
    • Deactivated the default guest account to prevent anonymous access and block the vulnerability used during exploitation net user guest /active:no
    • Viewed the administrative accounts to identify any unauthorized users that may have been created: net localgroup administrators and disabled the unauthorized hacker account created during the exploitation phase: net user hacker /active:no

    Outcome & Key Learnings:

    Exploitation is easy when systems are unpatched/misconfigured; Vulnerability of Clear-Text Protocols and Default Services was demonstarted; Effective hardening prevents exploitation; Importance of Best Practices: disabling the guest account alone was enough to prevent the MS17-010 exploit from running.

    Windows Registry Analysis

    Objective: To capture a live Windows Registry's hive files, analyze the live Windows Registry using regedit, analyze an offline FTK image of the Windows registry using Windows Registry Recovery (WRR).

    Methodology & Tools:

    • Identified the Windows Registry as a critical database for system forensics and malware analysis.
    • Used regedit command to view and edit registry keys which are associated with the system settings and the programs installed; viewed TCP/IP settings, startup programs, and language settings.
    • Obtained a Live Windows Registry with FTK Imager and saved a copy (backup) on disk for editing.
    • Analyzed the Registry Files with MiTeC Windows Registry Recovery (WRR.exe)
    • Inspected SAM file (Security Account Manager) - this file holds all the account informaon for the users and groups on the computer.
    • Viewed system Install Date, Startup Applications, Services and Drivers.
    • Inspected Network Configuration TCP/IP settings and User info NTUSER.DAT.

    Outcome & Key Learnings:

    Viewing a live registry is dangerous due risk of system corruption. A safer, better approach is to examine an offline registry. To do this, use FTK Imager to extract the files and then use MiTeC Windows Registry Recovery to analyze them. The MiTeC tool provides a much easier and more readable interface for viewing registry values compared to the standard regedit.

    Implementing Security Policies on Windows and Linux

    Objective: To implement basic security hardening measures on two operating systems: Microsoft Windows 10 and the Ubuntu (Linux) 8.04 virtual machines.

    Methodology & Tools:

      Securing Windows Logon and Implementing a Warning Banner:
    • Ran Netplwiz.exe to open the User Accounts login configuration.
    • Disabled the automatic login feature to require a username and password upon logon.
    • Opened the Group Policy Management Console using gpedit.msc.
    • Configured the Interactive logon: Message text policy to display "Unauthorized use is prohibited".
    • Configured the Interactive logon: Message title policy to display "Warning!" prior to the logon screen.

      • Auditing Logon Failures on Windows:
    • Simulated a failed login.
    • Opened Computer Management and viewed the Security log in Event Viewer and confirmed that "Audit Failure" logs were not being recorded by default.
    • Opened the Group Policy Management Console using gpedit.msc.
    • Enabled the Failure setting for the Audit account logon events in Audit Policy policy.
    • Signed out, attempted a failed login again, and logged back in.
    • Returned to the Security log and confirmed the new Audit Failure event was successfully logged.

      • Securing Linux Services and Firewall:
    • Ran the command zenmap (still on Windows machine) and scanned the Ubuntu machine to reveal all open ports.
    • Identified and observed insecure open services, such as Telnet (Port 23).
    • Used the command telnet to connect to Ubuntu.
    • Set the Linux kernel firewall's default input policy to DROP all incoming traffic using iptables -P INPUT DROP.
    • Added a specific rule to ALLOW web traffic on Port 80 with iptables -A INPUT -i eth0 -p tcp --dport 80 -j ACCEPT.
    • Scanned the Ubuntu host again from the Windows host using zenmap.
    • Confirmed that only Port 80 was now open, verifying the successful firewall hardening.

    Outcome & Key Learnings:

    Out-of-the-box OSs (both Windows and Linux) contain vulnerabilities that, if unpatched/reconfigured, pose a significant deficit in an organization's security posture.
